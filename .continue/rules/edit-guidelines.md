1. Always understand the full context before making code changes. Read related files if necessary.

2. Do not modify existing logic unless it is required for the requested change.

3. Prefer minimal and targeted changes instead of large refactors.

4. Do not remove existing functionality unless explicitly instructed.

5. Preserve existing coding style, naming conventions, and project structure.

6. If a change affects multiple files, explain why each file needs modification.

7. Never introduce breaking changes to APIs, database schemas, or interfaces without explicit confirmation.

8. When modifying database queries or schemas, ensure backward compatibility.

9. Do not hardcode secrets, credentials, API keys, or sensitive values.

10. Write clear and maintainable code. Avoid unnecessary complexity.

11. If unsure about a requirement or implementation, ask for clarification before changing code.

12. When suggesting improvements or optimizations, provide them separately from the main requested change.

13. Ensure that new code includes proper error handling and logging where appropriate.

14. Do not add new dependencies or libraries unless necessary. If added, explain the reason.

15. When changing existing functions, preserve their inputs and outputs unless explicitly instructed.

16. Prefer fixing the root cause instead of applying temporary workarounds.

17. If multiple solutions exist, briefly explain the alternatives and recommend the safest one.
